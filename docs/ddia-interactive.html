Reliability — Guvenilirlik

"Bir seyler ters gitse bile dogru calismaya devam etmek." Ters gidebilecek seylere fault (ariza), sistemi tamamen durdurmaya failure (cokme) denir. Amac: fault'lara ragmen failure'i onlemek — fault-tolerant tasarim.

Fault ≠ Failure

Fault: Bir bilesen beklenen davranistan saptiginda. Failure: Sistem butunuyle hizmet veremez hale gelmesi. Netflix'in Chaos Monkey'si production'da kasitli arizalar uretir ki sistem dayanikli olsun.

Donanim Arizalari

Sabit disklerin ortalama omru 10-50 yil (MTTF). 10.000 diskli kumede her gun 1 disk arizasi bekleyin. Geleneksel cozum: RAID, yedek guc, hot-swap. Modern cozum: yazilim duzeyinde tolerans — bulut platformlari VM'leri uyarisiz kapatabilir.

Yazilim Hatalari

Donanim hatalarinin aksine sistematik — bilesik arizalara yol acar. 2012'de leap second Linux kernel bug'i bircok uygulamayi ayni anda cokertmistir. Cascade failure: bir servis yavaslayinca downstream'ler de coker.

Insan Hatalari

Buyuk kesintilerin oncu sebebi operatorler — donanim sadece %10-25. Cozumler: hata firsatini azaltan API'ler, sandbox ortamlari, kapsamli test, hizli rollback, telemetri.

Scalability — Olceklenebilirlik

Artan yuk ile basa cikma kapasitesi. "X olceklenebilir" demek anlamsiz — dogru soru: "Sistem buyurse seçeneklerimiz neler?"

Twitter Fan-out Problemi

Tweet gonderme: Ortalama 4.6K istek/sn. Home timeline okuma: 300K istek/sn.

Yaklasim 1: Tweet'i global tabloya yaz, timeline okurken JOIN. 300K/sn'de JOIN pahali.
Yaklasim 2 (Fan-out): Her tweet'i takipcilerin cache kutularina yaz. 75 takipci × 4.6K = 345K yazma/sn. Okuma anlik.
Sorun: Unlulerin 30M+ takipcisi → 1 tweet = 30M yazma! Hibrit cozum: Normal kullanicilar icin fan-out, unluler icin okuma aninda birlestir.

Performansi Tanimlamak

Throughput: Saniyede islenen kayit sayisi (batch). Response time: Istegin gonderilip yanitin alinma suresi (online).

Latency ≠ Response Time

Latency: Istegin islenmeyi bekledigi sure (kuyruk). Response time: Istemcinin gordugu toplam sure (ag + kuyruk + islem). Response time latency'yi icerir.

Response time'i dagilim olarak dusunun — percentile ile olcun:

Percentile Nedir?

p50 (Medyan): Ortadaki deger — kullanicilarin yarisi bundan hizli hizmet aliyor.
p95: Her 20 istekten 1'i daha yavas. p99: Her 100'den 1'i. p99.9: Amazon bu degeri kullanir — en yavas yanit alan musteriler genellikle en degerli olanlardir (en cok veriye sahip).

Tail Latency Amplification

Bir son kullanici istegi birden fazla backend cagrisina neden olur. Tek yavas cagri tum istegi yavaslatir. 5 paralel backend cagrisinin herhangi biri p99'da ise son kullanicinin yavas istek orani ~%5'e cikar.

Yukle Basa Cikma

Scale up (dikey): Daha guclu makine. Scale out (yatay): Yuku cok makineye dagit. Pratikte ikisinin karisimi. Bazi sistemler elastiktir (otomatik kaynak ekleme), bazilari manueldir.

Maintainability — Bakilabilirlik

Yazilim maliyetinin cogu ilk gelistirmede degil devam eden bakimda. Uc ilke:

Operability

Isletim ekiplerinin hayatini kolaylastir: iyi izleme, otomasyon, tek makine bagimliligindan kacma, iyi varsayilanlar.

Simplicity

Karmasikligi yonet. "Camur topu" onle. En iyi arac: soyutlama — uygulama detayini gizle, temiz cephe sun.

Evolvability

Degisimi kolaylastir. Gereksinimler surekli degisir. Basitlik + iyi soyutlama = kolay evrim.

Relational vs Document Model

Relational Model (1970, Edgar Codd): Veriler satirlardan olusan tablolarda. Document Model: JSON/XML belgeleri, ic ice yapilar.

NoSQL'in Motivasyonlari

  • Cok yuksek yazma throughput / buyuk veri setleri
  • Acik kaynak tercihi
  • Ozellestirilmis sorgu islemleri
  • Daha dinamik ve ifade edici veri modelleri istegi

Object-Relational Mismatch

Ozgecmis Ornegi

Relational: users + positions + education tablolari + FK'lar + JOIN.
Document (JSON): Tum profil tek belge. Bagimli veriler ic ice. JOIN gereksiz, veri yerelligi iyi.

Sorgu Dilleri

Imperative: Nasil yapilacagini adim adim soyle (for, if). Declarative: Ne istedigini soyle, DB nasil yapacagina karar versin (SQL). Avantaj: optimizer otomatik iyilestirir, paralel calistirir.

SELECT name FROM people WHERE country = 'US' ORDER BY name;
db.people.find({ country: "US" }).sort({ name: 1 });
MATCH (p:Person)-[:LIVES_IN]->(:Country {name:'US'}) RETURN p.name ORDER BY p.name;

Graph Veri Modelleri

Many-to-many cok karmasiklastiginda graph dogal hale gelir.

Property Graph (Neo4j)

Vertex: ID + ozellikler + kenarlar. Edge: ID + kaynak + hedef + tur + ozellikler. Herhangi vertex herhangi vertexe baglanabilir.

Triple-Store (SPARQL)

Tum bilgi (subject, predicate, object) uc'luleri. Ornek: (Lucy, age, 33) veya (Lucy, livesIn, Istanbul).

Bolum 3

Depolama ve Erisim

Veritabani veriyi nasil sakliyor? Geri istediginde nasil buluyor? Her gelistirici bunu bilmeli.

En Basit DBLSM-TreeB-TreeLSM vs B-TreeOLTP vs OLAPColumn Storage

Dunyanin En Basit Veritabani

db_set() { echo "$1,$2" >> database; } db_get() { grep "^$1," database | sed "s/^$1,//" | tail -n 1; }

db_set — dosya sonuna ekle: O(1). db_get — dosyayi tara: O(n). Temel trade-off: index okumalari hizlandirir ama her indeks yazmalari yavaslatir.

SSTable ve LSM-Tree

SSTable (Sorted String Table): Anahtarlari sirali tutan log segmenti. Avantaj: merge-sort ile birlestirme O(n), sparse index yeter (tum anahtarlari tutma), range query dogal.

LSM-Tree Yazma Yolu

0/5

Kullanan sistemler: LevelDBRocksDBCassandraHBaseLucene (Elasticsearch)

Bloom Filter: Bellekte tutulan olasiliksal yapi — "bu anahtar bu segmentte yok" kesin soylenir, gereksiz disk okumalari onlenir.

B-Tree

En yaygin indeks yapisi (1970'ler). Sabit boyutlu sayfa (page, 4KB) tabanli — her seferde bir sayfa okunur/yazilir. Diske donuk isaretciler ile agac yapisi.

B-Tree Arama

Crash Guvenligi

B-Tree sayfalari yerinde gunceller — crash'te yarim yazilmis sayfa riski. Cozum: WAL (once log'a yaz, sonra sayfaya). Ayrica latch (hafif kilit) ile esanli erisim korumasi.

OLTP vs OLAP

OLTP

Kucuk islemler, anahtar ile random erisim, kullanici girdisi, GB-kucuk TB, darbogazı: disk seek. Ornek: e-ticaret, banka, bilet satis.

OLAP

Cok kayit uzerinde aggregate (SUM, COUNT), ETL ile bulk import, TB-PB, darbogazı: disk bandwidth. Ornek: gelir raporu, kullanici davranis analizi.

Data Warehouse: OLTP'den veri kopyala (ETL), analitik sorgulari burada calistir. RedshiftBigQuerySnowflake

Column-Oriented Storage

OLAP sorgusu 100+ sutunlu tablonun 3-5 sutununa ihtiyac duyar. Row storage TUM sutunlari okur. Cozum: Her sutunu ayri dosyada sakla — yalnizca gerekli sutunlar okunur. Bitmap encoding: Tekrarlanan degerleri bit dizisiyle sikistir. Materialized View / Data Cube: Sik kullanilan aggregate'ler onceden hesaplanip saklanir.

Protobuf/Thrift: Tag Numaralari

Her alan bir tag numarasi (1,2,3...) ve tip ile tanimlanir. Kodlanmis veride alan adi yok, sadece tag var. Alan adini degistir ama tag numarasi asla degismemeli.

Avro: Yazici ve Okuyucu Semalari

Kodlanmis veride tag bile yok — sadece degerler sirali. Writer schema ve reader schema arasindaki fark cozulur. Hadoop ekosisteminin standart formati.

Schema Evolution

Forward Compatibility

Eski kod yeni kodun yazdigini okuyabilir. Nasil: bilinmeyen tag'lari atla.

Backward Compatibility

Yeni kod eski kodun yazdigini okuyabilir. Nasil: yeni alanlara varsayilan deger.

Veri Akis Modlari

Veritabani Uzerinden

Yazma encode, okuma decode. DB uzun yillar kalici — data outlives code. 5 yil onceki format okunabilmeli.

REST / RPC

REST: HTTP + kaynak tabanli. RPC: Uzak fonksiyon cagrisini yerel gibi goster (gRPC, Thrift). Sorun: ag cagrilari lokal cagrilardan temelden farkli.

Asenkron Mesaj

Broker uzerinden: RabbitMQKafka. Tampon, yeniden deneme, alici cevrimdisi olabilir.
