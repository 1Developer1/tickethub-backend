Renk Kodlari & Ok Turleri

Her post-it rengi farkli bir anlam tasir. Oklar ise "kim kimi tetikliyor, nasil" sorusunu yanitlar.

Turuncu — Domain Event
"Bir sey OLDU" (gecmis zaman). Sistemdeki olgular.
Ornek: "Bilet Rezerve Edildi", "Odeme Alindi"
Mavi — Command
"Bir sey YAP" (emir kipi). Niyet, istek.
Ornek: "Koltugu Sec", "Odemeyi Yap"
Sari — Actor / User
"Kim yapiyor?" Komutu veren insan.
Ornek: "Musteri", "Organizator", "Admin"
Mor — Policy / Kural
"X olunca OTOMATiK Y yap". Insan tetiklemez, sistem tetikler.
Ornek: "10 dk dolunca → expire et"
▼ Asagida detayli aciklama var
Yesil — Read Model
"Kullaniciya ne gosteriyoruz?"
Ornek: "Musait koltuk haritasi", "Fiyat gecmisi"
Pembe — External System
"Dis servis" — bizim kontrolumuzde degil.
Ornek: Stripe, SMS, Email servisi
Kirmizi — Hot Spot
"Sorunlu / belirsiz / tartismali / riskli nokta"
Ornek: "Ayni koltuğa 2 kisi tiklarsa?"

Dis servis cagrisi

Mor Post-it (Policy) Nedir?

Tek cumleyle: "Bir event olunca, INSAN MUEDAHALESI OLMADAN, otomatik calisan kural."

Normal akis: Insan bir sey yapmak ister → Command gonderir → Event olusur.
Policy akisi: Event olusur → Policy OTOMATIK tetiklenir → yeni Event veya Command olusur.

Aradaki fark: policy zincirinde kullanici YOK. Sistem kendi kendine karar veriyor. Bu yuzden ok KESIKLI ciziliyor — "bu adim otomatik, biri tiklamadi" anlami tasiyor.

👤 Insan Tetikli Akis (DUZ ok)

Musteri ➔ Koltugu Sec ➔ Koltuk Secildi
Musteri bilinçli tikliyor → komut gonderiyor → event olusuyor.
Her adimda bir INSAN karari var.

⚡ Policy Tetikli Akis (KESIKLI ok)

Koltuk Secildi ⇢ 10dk zamanlayici kur ⇢ Suresi Doldu
Kimse tiklamadi. Event oldu → sistem otomatik kurali calistirdi → yeni event oldu.
BullMQ delayed job, cron, veya if-then kurali.

TicketHub'daki Tum Policy'ler

🔸 WHEN: Koltuk secildi (hold olusturuldu)
⚡ THEN: 10 dakika sonra expire et
→ BullMQ delayed job (600.000ms)
🔸 WHEN: Odeme basariyla alindi
⚡ THEN: Rezervasyonu onayla
→ reservation.confirmPayment()
🔸 WHEN: Rezervasyon onaylandi
⚡ THEN: QR bilet uret + email gonder
→ asyncEventBus.emit() × 2
🔸 WHEN: Rezervasyon olusturuldu (talep degisti)
⚡ THEN: Fiyati yeniden hesapla
→ pricingService.applySurge()
🔸 WHEN: Kalan koltuk < %20
⚡ THEN: Surge fiyat uygula (x1.3)
→ PRICING_RULES.LOW_AVAILABILITY
🔸 WHEN: Bilet iptal edildi + onaylanmisti
⚡ THEN: Odemeyi iade et
→ paymentsService.refund()

Policy = kodda nereye denk dusuyorlar?
• BullMQ delayed job (zamanlayici policy'ler) → asyncEventBus.emit('...', payload, { delay: 600000 })
• Sync event handler (aninda policy'ler) → syncEventBus.on('reservation.created', handler)
• Entity icindeki if-then (is kurali policy'ler) → if (status !== 'PENDING') throw

DDD Yapi Taslari Building Blocks — Entity, Value Object, Aggregate Root, Domain Event, Domain Service

Event Storming'de post-it'leri duvara yapistirdik. Simdi bu post-it'ler hangi kod yapisina donusuyor? Bu bolumde her bir DDD yapi tasini, ne ise yaradigini, nasil yazildigini ve projemizde nerede kullanildigini goreceksiniz — Node.js, C#, Java dillerinde.

1. Value Object

Ne? Kimligi olmayan, degeri ile tanimlanan nesne. 100 TL = 100 TL — iki farkli nesne ama esit.

⚠ Immutable Olmazsa Ne Olur? (Yan Etkiler)

Yan etki (side effect): Bir nesneyi degistirdiginizde, o nesneye referans tutan baska kodun haberi olmadan etkilenmesidir.

❌ MUTABLE Money — Bug!
// reservation ve receipt AYNI Money objesini paylasiyor
const price = new Money(5000);
reservation.totalPrice = price;
receipt.amount = price;

// Sonra biri KDV ekliyor...
price.amount += 900; // ← MUTATE!

// Simdi reservation.totalPrice = 5900 💥
// receipt.amount = 5900 💥
// IKISI DE degisti — haberleri olmadan!
✅ IMMUTABLE Money — Guvenli
// reservation ve receipt AYNI Money objesini paylasiyor
const price = Money.fromCents(5000);
reservation.totalPrice = price;
receipt.amount = price;

// KDV ekleniyor — YENI nesne doner
const withTax = price.add(Money.fromCents(900));

// price hala 5000 ✅
// reservation.totalPrice hala 5000 ✅
// receipt.amount hala 5000 ✅
// withTax = 5900 (yeni nesne)
🔢 Aritmetik ve Karsilastirma "Icsel Olarak Tanimlanir" Ne Demek?

Icsel (encapsulated): Para islemi Money sinifinin icinde tanimlanir — disarida carpma bolme yapilmaz. Cunku disarida yaparsan kurallari atlama riski var:

❌ Disarida Aritmetik — 3 farkli bug
// Bug 1: Float hesaplama
const total = 19.99 * 3; // 59.97000000000001 💥

// Bug 2: Farkli para birimi toplama
const x = tryPrice + usdPrice; // TRY + USD? 💥

// Bug 3: Negatif sonuc
const r = 100 - 500; // -400 cent? 💥
✅ Icsel Aritmetik — kurallar korunur
// Toplama: para birimi kontrol edilir
price.add(tax);   // ikisi de TRY ise OK
price.add(usd);   // HATA: "Farkli para birimi"

// Cikarma: negatif sonuc engellenir
small.subtract(big); // HATA: "Negatif olamaz"

// Carpma: integer yuvarlatilir
price.multiply(1.18); // Math.round → tam sayi cent

Karsilastirma da ayni mantik: price.equals(other) hem miktari hem para birimini kontrol eder. price.isGreaterThan(other) farkli para birimlerini reddeder. Disarida a.amount > b.amount yaparsan para birimini kontrol etmezsin.

Ozellik 1
Immutable (degistirilemez)
Olusturulduktan sonra hicbir alan degismez — paylasilan referanslarda yan etki olmaz
Ozellik 2
Esitlik = icerik
ID yok, tum alanlari ayni = esit. equals() metodu ile kontrol
Ozellik 3
Self-validating
Constructor'da kurallar zorlanir: negatif ×, float ×, farkli para birimi ×
Ozellik 4
Icsel aritmetik + karsilastirma
add(), subtract(), multiply(), equals(), isGreaterThan() — hepsi sinif icinde, kurallarla
💡 Value Object'in Kurallarini Nasil Belirleriz?

Kendine 3 soru sor — cevaplar kurallarin kendisi:

Soru 1: Bu deger tek basina gecerli mi?

Degerin kendi basina anlamsiz veya gecersiz olabilecegi durumlari listele. Her "Hayir" cevabi constructor'da bir invariant olur:

Money icin sorgula:
Negatif olabilir mi?    → Hayir  → kural
Ondalikli olabilir mi?  → Hayir  → kural
Para birimi bos?        → Hayir  → kural
Sifir olabilir mi?      → Evet   → kural YOK
"Hayir"lar → constructor kurallari:
private constructor(cents: number, cur: string) {
  if (cents < 0)
    throw 'Negatif olamaz';
  if (!Number.isInteger(cents))
    throw 'Tam sayi olmali';
  if (!cur?.trim())
    throw 'Para birimi bos';
}
Soru 2: Bu deger uzerinde hangi islemler yapilir?

Gercek dunyada bu degerle ne yapiyorsun? Her islemin kendi kurali var:

Gercek dunya islemleri:
Iki parayi topluyorum         → add()
Birinden cikariyorum          → subtract()
Carpiyorum (KDV, indirim)     → multiply()
Karsilastiriyorum             → equals()
Kullaniciya gosteriyorum      → toDisplayString()
Her islemin kurali:
add(other) {
  // kural: farkli birim toplanamaz
  if (this.cur !== other.cur) throw...
  return new Money(this.c + other.c);
}
subtract(other) {
  // kural: sonuc negatif olamaz
  if (this.c - other.c < 0) throw...
}
multiply(factor) {
  // kural: tam sayiya yuvarla
  return new Money(Math.round(...));
}
Soru 3: Bu degerin esitligi neye gore?

Value Object'te ID yok — esitlik tum alanlarin esitligidir:

100 TRY === 100 TRY?  → Evet  (miktar + birim ayni)
100 TRY === 100 USD?  → Hayir (birim farkli)

equals(other: Money): boolean {
  return this._amountInCents === other._amountInCents
      && this._currency === other._currency;
}

Soru 1: Gecerlilik

Soru 2: Islemler

Soru 3: Esitlik

Kural disaridan gelmiyor — gercek dunyada bu degerin nasil davrandigini dusun, cevaplar kurallarin kendisi.

📏 Metafor: Olcum Aleti

Value Object bir olcum aletidir — cetvel, termometre, tarti gibi. 30 cm'lik bir cetvel dusun: kimligi yok (seri numarasiyla ayirt etmezsin), degismez (yarin 35 cm olmaz), iki 30 cm cetvel birbirinin aynisi. 30 cm + 20 cm = yeni bir 50 cm cetvel — eski cetvel hala 30 cm. Acip kapatmazsin, icine bir sey takmazsin, durumu degismez.

❓ Neden Ekstra Nesne Uretelim — Duz Sayi Yetmez mi?

Tek basina calisiyorsan fark etmez. 5+ kisilik projede hayat kurtarir:

❌ Duz sayi — 50 gelistirici 50 farkli yapar
// Gelistirici A: float bug
const total = 19.99 * 3; // 59.97000000000001

// Gelistirici B: farkli birim topladi
const x = tryPrice + usdPrice; // TRY+USD?

// Gelistirici C: negatif kontrol unuttu
const r = 100 - 500; // -400 cent

// Gelistirici D: userId'yi para sanip gonderdi
chargeFee(userId); // ikisi de number!
✅ Money sinifi — kural bir yerde, 50 yerde korunur
// Float bug imkansiz
price.multiply(3); // Math.round otomatik

// Farkli birim engellenir
tryPrice.add(usdPrice); // HATA!

// Negatif engellenir

// Tip guvenligi

Sistemde yuzlerce yerde para degeri var (reservation.totalPrice, seatHold.price, payment.amount, refund.amount...). Hepsi ayni sey — ayni turden deger cok kez geciyorsa ve kurallari varsa → Value Object yap, bir kez yaz, her yerde kullan.

🔒 readonly ile Immutability Garantisi

Heap'deki kutuyu kimsenin degistirememesi icin alanlar readonly isaretlenir:

private constructor(
  private readonly _amountInCents: number,   // readonly = yazilamaz
  private readonly _currency: string,        // readonly = yazilamaz
) { }

this._amountInCents = 5900;  // ❌ DERLEME HATASI — readonly

C#'ta record kullanilir (otomatik immutable). Java'da record kullanilir (otomatik immutable). TypeScript'te readonly elle yazilir.

🔗 Value Object ile Schema (Zod) Iliskisi

Farkli katmanlarda farkli seyleri kontrol ederler:

Schema (Zod) — kapida
"priceInCents bir sayi mi?" — tipi kontrol eder
Value Object (Money) — iceride
"sayi geldi ama negatif mi?" — anlami kontrol eder

Schema'dan gecen veri hala gecersiz olabilir — schema z.number() ile sayiyi gecirdi ama deger -500 olabilir. Money.fromCents(-500) bunu yakalar.

Projede: src/modules/booking/domain/money.value-object.ts, seat-hold.value-object.ts

// Node.js / TypeScript — Value Object: Money
export class Money {
  // private constructor → disaridan new Money() yapilamaz
  private constructor(
    private readonly _amountInCents: number,
    private readonly _currency: string = 'TRY',
  ) {
    // Invariant: negatif para olamaz
    if (_amountInCents < 0) throw new Error('Negatif para olamaz');
    // Invariant: tam sayi olmali (kurus = en kucuk birim)
    if (!Number.isInteger(_amountInCents)) throw new Error('Tam sayi olmali');
  }

  // Factory method — tek giris noktasi
  static fromCents(amount: number, currency = 'TRY'): Money {
    return new Money(amount, currency);
  }

  // Aritmetik → her zaman YENI Money doner (immutable)
  add(other: Money): Money {
    this.assertSameCurrency(other);
    return new Money(this._amountInCents + other._amountInCents, this._currency);
  }

  // Esitlik → icerige gore (ID yok!)
  equals(other: Money): boolean {
    return this._amountInCents === other._amountInCents
        && this._currency === other._currency;
  }
}
// C# — Value Object: Money (record ile dogal immutable)
public sealed record Money
{
    public int AmountInCents { get; }
    public string Currency { get; }

    // Private constructor — disaridan new Money() yapilamaz
    private Money(int amountInCents, string currency)
    {
        if (amountInCents < 0) throw new ArgumentException("Negatif para olamaz");
        AmountInCents = amountInCents;
        Currency = currency;
    }

    // Factory method
    public static Money FromCents(int amount, string currency = "TRY")
        => new(amount, currency);

    // Aritmetik → yeni Money (record zaten immutable)
    public Money Add(Money other)
    {
        AssertSameCurrency(other);
        return new(AmountInCents + other.AmountInCents, Currency);
    }

    // record: Equals ve GetHashCode otomatik icerik bazli uretilir
}
// Java — Value Object: Money (record ile dogal immutable, Java 16+)
public record Money(int amountInCents, String currency) {

    // Compact constructor — validation
    public Money {
        if (amountInCents < 0) throw new IllegalArgumentException("Negatif para olamaz");
        if (currency == null) throw new NullPointerException();
    }

    // Factory method
    public static Money fromCents(int amount) {
        return new Money(amount, "TRY");
    }

    // Aritmetik → yeni Money
    public Money add(Money other) {
        assertSameCurrency(other);

// Node.js / TypeScript — Aggregate Root: Reservation
// Her command metodu 4 adim izler:

confirmPayment(paymentId: string): void {
  // 1. GUARD — kurallari kontrol et, ihlal varsa hata firlat
  if (this.props.status !== 'PENDING')
    throw new Error(`'${this.props.status}' durumunda onaylanamaz`);
  if (this.isHoldExpired())
    throw new Error('Hold suresi dolmus — koltuklar serbest birakildi');

  // 2. MUTATE — durumu degistir
  this.props.status = 'CONFIRMED';
  this.props.confirmedAt = new Date();
  this.props.paymentId = paymentId;

  // 3. VERSION — optimistic concurrency icin
  this.props.version += 1;

  // 4. EVENT — ne oldu kaydet (yan etkiler icin)
  this._domainEvents.push({
    type: 'reservation.confirmed',
    reservationId: this.props.id,
    userId: this.props.userId,
    paymentId,
  });
}

cancel(): void {
  // GUARD — gecersiz gecisler
  if (this.props.status === 'EXPIRED')
    throw new Error('Suresi dolmus rezervasyon iptal edilemez');

  // GUARD — is kurali: etkinlige 48+ saat kalmali
  if (this.props.status === 'CONFIRMED') {
    const hours = this.hoursUntilEvent();
    if (hours < 48)
      throw new Error(`Etkinlige ${hours}h kaldi. Minimum 48h`);
  }

  // MUTATE + VERSION + EVENT
  this.props.status = 'CANCELLED';
  this.props.version += 1;
  this._domainEvents.push({ type: 'reservation.cancelled', /* ... */ });
}

🚦 Durum Gecisleri Nasil Yonetilir? (State Machine)

Her command metodu bir durum gecisidir. Hangi gecislere izin verilir, hangileri yasaktir — bunu guard clause'lar belirler:

Her ok bir command metodu. Her metodun icindeki guard yasak gecisleri engeller:

// Durum gecis kurallari — her metod kendi gecislerini bilir

confirmPayment(paymentId) {
  // IZIN VERILEN: PENDING → CONFIRMED
  if (status !== 'PENDING') throw 'Sadece PENDING onaylanabilir';
  // EK KURAL: zaman kisitlamasi
  if (this.isHoldExpired()) throw '10 dk doldu, koltuklar serbest';
  // Gecis
  status = 'CONFIRMED';
}

expire() {
  // IZIN VERILEN: PENDING → EXPIRED
  if (status !== 'PENDING') return; // sessiz — race condition korunmasi
  // Neden throw degil return?
  // confirm + expire ayni anda calisabilir (BullMQ + HTTP)
  // confirm kazandiysa expire sessizce geri donmeli
  status = 'EXPIRED';
}

cancel() {
  // IZIN VERILEN: PENDING → CANCELLED, CONFIRMED → CANCELLED
  // YASAK: EXPIRED veya zaten CANCELLED
  if (status === 'EXPIRED')   throw 'Suresi dolmus, iptal edilemez';
  if (status === 'CANCELLED') throw 'Zaten iptal edilmis';

  // EK KURAL: CONFIRMED ise etkinlige 48+ saat kalmali
  if (status === 'CONFIRMED') {
    if (this.hoursUntilEvent() < 48)
      throw 'Etkinlige 48 saatten az kaldi';
  }
  status = 'CANCELLED';
}

"Bu durumdayken su duruma gecebilir mi?"
EXPIRED → CONFIRMED yasak. PENDING → CONFIRMED izinli.
Kural Turu 2: Is Kurali
"Gecise izin verilse bile ek kosullar saglanmali mi?"
CONFIRMED → CANCELLED izinli AMA etkinlige 48+ saat kalmali.
Kural Turu 3: Zaman Kurali
"Islem zamani uygun mu?"
PENDING → CONFIRMED izinli AMA hold suresi (10 dk) dolmamis olmali.
🔒 Version Neden Arttirilir? (Optimistic Concurrency)

Iki HTTP istegi ayni anda ayni reservation'i degistirmeye calisirsa ne olur?

❌ Version olmadan — ikisi de kazanir
// Istek A: confirm
UPDATE SET status='CONFIRMED' WHERE id=:id
// ✅ basarili

// Istek B: cancel (ayni anda)
UPDATE SET status='CANCELLED' WHERE id=:id
// ✅ basarili — CONFIRMED'i ezdi!
// 💥 Para alindi ama rezervasyon iptal
✅ Version ile — sadece biri kazanir
// Istek A: confirm (version 1 → 2)
UPDATE SET status='CONFIRMED', version=2
  WHERE id=:id AND version=1
// ✅ 1 satir etkilendi — basarili

// Istek B: cancel (version hala 1 bekliyor)
UPDATE SET status='CANCELLED', version=2
  WHERE id=:id AND version=1
// ❌ 0 satir etkilendi — baskasi araya girdi!
// → ConflictError firlat, istemci tekrar denesin
❗ Root'u Ozel Yapan Sadece Durum Degistirmesi mi?

Hayir. Entity de durum degistirir. Root'u ozel yapan uc seyin birlikte olmasi:

  1. Tek giris noktasi — icindeki nesnelere disaridan sadece root uzerinden erisim
  2. Kume yonetimi — icindeki nesnelerin tutarliligini koordine eder (SeatHold eklenince totalPrice degisir)
  3. Yasam dongusu — DB'den yukle, islet, geri kaydet (islem makinesi)

Entity = makinenin icindeki motor (kendi basina donebilir ama makine acmadan donduremezsin). Root = makinenin kendisi.

📚 Aggregate Root'un Kesin Tanimi

"Birlikte degismesi gereken nesnelerin kumesini tek noktadan yoneten, disariya tek giris noktasi olan, her durum degisikliginde kurallarini kontrol eden, icindeki tum nesnelerin tutarliligini garanti eden entity."

🔌 Her Entity Icin Root Gerekli mi?

Hayir. Bir entity 3 sekilde var olabilir:

1. Kendi basina root
En yaygin. User, Venue, Reservation. Kendi kumesinin lideri.
2. Bir root'un icinde
OrderLine gibi. Kendi ID'si var ama root uzerinden yonetilir. Root ile tutarlilik bagimliligi var.
3. Bagimsiz (pratikte yok)
Teorik. Kurallarini kim koruyacak? Ya kendi root'u olur ya baskasinin icine girer.
📊 Root Bellekte Ne Kadar Yer Kaplar ve Ne Kadar Yasarr?

Bir Reservation ~500 byte. 10.000 esanli istek olsa: 5 MB. Sunucunun 8 GB RAM'inde hicbir sey. Bellek sorunu degil.

Root zamaninin %98'ini bekleyerek geciyor:

0ms        root olusturuldu (fromPersistence)
0.1ms      nesne bellekte hazir (500 byte)
           ... DB sorgusunu bekliyor ...     ← zamanin %90'i
8ms        DB'den veri geldi
8.1ms      guard + mutate + version++        ← 0.1ms (nanosaniyeler)
           ... DB'ye yazmayi bekliyor ...    ← zamanin %90'i
15ms       DB'ye yazildi
15.01ms    nesne oldu (GC temizler)

Root'un kendi isi: 0.2ms  |  DB bekleme: 14.8ms  |  Toplam: ~15ms

Root hafif bir zarftir — icindeki is ucuz, zarfi postalama (DB'ye gidip gelme) pahali.

3.5 Factory Method (Neden Private Constructor?)

Tanim: Ayni nesneyi farkli kurallara gore olusturmak gerektiginde, olusturma mantığını ayri metodlara ayirma teknigi.

🔑 Constructor Neden Private?

Nesnenin gecersiz bir sekilde olusturulmasini engellemek icin — olusturma her zaman kontrol edilmis bir factory method'dan gecsin diye. Ama asil sebep daha derin:

❌ Constructor public + icinde kural
public constructor(props) {
  // create() icin dogru:
  if (props.expiresAt <= new Date())
    throw 'Gecmis tarih';
  if (props.status !== 'PENDING')
    throw 'PENDING olmali';

// fromPersistence PATLADI!
fromPersistence({
  status: 'CONFIRMED',    // ❌ "PENDING olmali"
  expiresAt: gecmisTarih  // ❌ "Gecmis tarih"
})
// DB'deki gecerli kayit yuklenemiyor!
✅ Constructor private + bos, kurallar factory'de
private constructor(props) {
  // BOS — kural yok
}

static create(props) {
  // TUM kurallar burada
  if (props.expiresAt <= new Date()) throw..
  return new Reservation(props);
}

static fromPersistence(props) {
  // KURAL YOK — DB verisi gecerli
  return new Reservation(props);  // ✅
}

Constructor public olsaydi ucuncu bir yol olurdu: new Reservation({...}) — factory'lerin kurallarini atlayarak nesne olusturma. Private olunca derleyici bunu engeller.

📑 Iliskili Pattern: Specification

Bir is kuralini ayri bir nesne olarak tanimlamak. Ayni kural birden fazla yerde kullaniliyorsa cikarilir:

❌ Ayni kural 3 yerde tekrar

  if (status === 'EXPIRED') throw...
  if (hoursUntil < 48) throw...
}
// 2. Admin panelinde

  if (r.status === 'EXPIRED') return false;
  if (r.hoursUntil < 48) return false;

// 3. Raporda

  // ayni if'ler UCUNCU KEZ
})
✅ Specification — kural bir kez
class CancellationSpec {
  isSatisfiedBy(r): boolean {
    if (r.status === 'EXPIRED') return false;
    if (r.hoursUntil < 48) return false;
    return true;
  }
}

// 3 yerde ayni nesne kullanilir:
cancel() { if (!spec.isSatisfiedBy(this)) throw... }
canShowBtn(r) { return spec.isSatisfiedBy(r); }
list.filter(r => spec.isSatisfiedBy(r));

Specification'lar and(), or(), not() ile birlestirilir: new IsActive().and(new BelongsToUser(id)).and(new IsExpiringSoon().not()). Kural 1-2 yerde kullaniliyorsa entity'de kalsin. 3+ yerde tekrarlaniyorsa Specification cikar.

        if (occupancy > 0.8)  return section.getBasePrice().multiply(1.3);
        return section.getBasePrice();
    }
}
