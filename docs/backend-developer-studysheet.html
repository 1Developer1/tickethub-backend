Backend Yazilim Gelistirmeye Giris Introduction to Backend Development

Backend = kullanicinin goremediği her sey. Tarayicidaki buton frontend, butona basinca olan is mantigi backend.

Frontend vs Backend

Frontend: HTML, CSS, JavaScript — kullanicinin gordugu ekran.
Backend: sunucuda calisan kod — is kurallari, DB islemleri, guvenlik.

Backend Ne Yapar?

Istegi al → dogrula → is kuralini calistir → DB'ye yaz/oku → cevap don.
Her sey bu dongunun varyasyonu.

Backend Dilleri

C#/.NET Java/Spring Node.js Python/Django Go PHP/Laravel
Dil degisir, mimari ayni kalir.

HTTP Istek-Cevap Dongusu

Client → POST /api/orders → Sunucu isler → 201 Created → Client cevabi alir.
Her web uygulamasi bu dongude calisir.

Kullanici (Tarayici/Mobil) | | HTTP istegi (POST /api/reservations) v ┌──────────────────────────────────────────────────┐ │ BACKEND SUNUCU │ │ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ │ │ Route │→ │ Service │→ │ Repository │ │ │ │ (giris) │ │ (is kurali)│ │ (DB erisim)│ │ │ └────────────┘ └────────────┘ └────────────┘ │ │ | │ │ v │ │ ┌──────────┐ │ │ │ Database │ │ │ └──────────┘ │ └──────────────────────────────────────────────────┘ | | HTTP cevap (201 Created + JSON) v Kullanici cevabi gordu

Neden Backend Onemli?

Frontend guzel gosterir, backend gercekten calistirir. Odeme islemi, stok kontrolu, guvenlik — hepsi backend. Frontend olmadan cirkin ama calisan bir sistem olur. Backend olmadan hicbir sey calismaz.

API nedir? Application Programming Interface — iki programin konusma protokolu. Backend API yazar, frontend (veya mobil) bu API'yi cagirir.

REST API

GET /users → kullanicilari getir
POST /users → yeni kullanici olustur
PUT /users/1 → guncelle
DELETE /users/1 → sil
HTTP metodlari ile CRUD islemleri.

JSON

Backend ve frontend arasinda veri formati:
{ "name": "Ahmet", "age": 25 }
Hemen her dil JSON okuyup yazabilir.

// C# .NET Core — basit bir endpoint
[ApiController]
[Route("api/[controller]")]
public class UsersController : ControllerBase
{
    private readonly UserService _service;

    [HttpGet]
    public async Task<IActionResult> GetAll()
    {
        var users = await _service.GetAllAsync();
        return Ok(users);
    }

    [HttpPost]
    public async Task<IActionResult> Create(CreateUserRequest request)
    {
        var user = await _service.CreateAsync(request);
        return CreatedAtAction(nameof(GetAll), user);
    }
}

Girdi alir → isler → cikti verir.

Veri Yapilari ve Temel Algoritmalar Data Structures & Algorithms

Veri yapisi = veriyi nasil organize edersin. Algoritma = o veriyle ne yaparsin.

Temel Veri Yapilari

Array (Dizi)

Sabit boyutlu, sirali eleman listesi.
Erisim: O(1) — index ile aninda ulas.
Ekleme/silme: O(n) — kaydirma gerekir.
[10, 20, 30, 40, 50]

Linked List (Bagli Liste)

Her eleman bir sonrakini gosterir.
Ekleme/silme: O(1) — referans degistir.
Erisim: O(n) — bastan say.
10 → 20 → 30 → null

Stack (Yigin) — LIFO

Son giren ilk cikar. Tabak yigini gibi.
push() → uste ekle
pop() → ustten al
Kullanim: geri al (undo), fonksiyon cagirilari.

Queue (Kuyruk) — FIFO

Ilk giren ilk cikar. Banka sirasi gibi.
enqueue() → sona ekle
dequeue() → bastan al
Kullanim: BullMQ job kuyrugu, istek sirasi.

Hash Map (Sozluk)

Anahtar-deger cifti. Aninda erisim.
Erisim: O(1)
{ "ahmet": 25, "mehmet": 30 }
Kullanim: cache, EventBus handlers, index.

Tree (Agac)

Hiyerarsik yapi. Her dugumun cocuklari var.
Binary Search Tree: sol < kok < sag
Arama: O(log n)
Kullanim: DB indexleri (B-Tree), dosya sistemi.

Big O — Karmasiklik Analizi

Algoritmanin ne kadar yavas veya hizli oldugunu olcer. Veri buyudukce nasil etkilenir?

NotasyonAnlamiOrnek1000 eleman
O(1)Sabit — veri buyuse de ayniArray'den index ile okuma1 islem
O(log n)Logaritmik — her adimda yari elerBinary search~10 islem
O(n)Dogrusal — her eleman icin 1 islemListe tarama1000 islem
O(n log n)Etkin siralamaMerge sort, Quick sort~10.000 islem
O(n²)Kareli — ic ice donguBubble sort1.000.000 islem

Temel Algoritmalar

Arama — Linear vs Binary

Linear: Bastan sona tara → O(n)
Binary: Sirali dizide ortadan bol → O(log n)
1000 elemanda: 1000 vs 10 islem.

Siralama

Bubble Sort: Yavas O(n²) — sadece ogretim icin
Merge Sort: Hizli O(n log n) — bol, sirala, birlestir
Quick Sort: Pratikte en hizli — pivot sec, etrafinda bol

Recursion (Ozyineleme)

Fonksiyonun kendini cagirmasi.
factorial(5) = 5 × factorial(4)
Her recursion'in bir durma kosulu olmali, yoksa stack overflow.

Mulakatlarda Ne Sorulur?

"Bu algoritmanin Big O'su nedir?", "Array mi Linked List mi kullanirsin?", "HashMap nasil calisir?" — bunlar klasik sorular. Ezberden degil mantikla cevap ver.

Isletim Sisteminin Temelleri Operating System Fundamentals

Isletim sistemi (OS) = donanim ile yazilim arasindaki tercuman. Programlar dogrudan CPU/RAM'e erisemez, OS araciligi ile erisir.

Process (Surec)

Calisan bir program. Her process'in kendi bellegi var, birbirine eriseemez.
Chrome, Node.js, PostgreSQL → her biri ayri process.

Thread (Is Parcacigi)

Process'in icindeki calisma birimi. Ayni bellegi paylasir.
Java: istek basina 1 thread.
Node.js: tek thread + await.

Process vs Thread

Process: Izole bellek, biri cokerse digeri etkilenmez, agir.
Thread: Paylasimli bellek, biri cokerse process coker, hafif.

CPU Scheduling

CPU tek seferde bir is yapar. OS hangi process/thread'in ne zaman calisacagina karar verir. Saniyede binlerce kez degistirir → coklu gorev yanilsamasi.

Bellek (RAM) Yonetimi

Bellek Yapisi: ┌──────────────────────────┐ │ Stack │ ← fonksiyon cagirilari, yerel degiskenler │ (kucuk, hizli, otomatik) │ fonksiyon bitince otomatik temizlenir ├──────────────────────────┤ │ Heap │ ← nesneler, buyuk veriler │ (buyuk, yavas, manuel/GC)│ GC (Garbage Collector) temizler └──────────────────────────┘

Eskamanlilik Sorunlari

Race Condition

2 thread ayni anda ayni veriye yazinca veri bozulur.
Ornek: Bakiye 1000 TL, 2 thread ayni anda 500 TL cekiyor → bakiye 500 gozukuyor, 500 TL kayip.
Node.js'te YOK — tek thread.

Deadlock

2 thread birbirini sonsuza kadar bekler.
A kilit X'i aldi, Y'yi bekliyor. B kilit Y'yi aldi, X'i bekliyor.
Ikisi de sonsuza kadar bekler → program dondu.
Node.js'te YOK — tek thread, kilit yok.

Backend Gelistirici Icin Ozet

Bilmen gereken: Process/Thread farki, race condition/deadlock nedir, bellek stack/heap ayrimi.
Bilmene gerek yok: CPU scheduler nasil yazilir, page fault handling, semaphore implementasyonu. Bunlar OS muhendisinin isi.

Domain-Driven Design (DDD)

Karmasik is kurallarini modellemek icin kullanilan tasarim yaklasimi:

Value Object

Kimligi yok, degeri ile tanimlanir, degismez.
Money(100, "TRY") — 100 TL = 100 TL

Entity

Kimligi var, durumu degisir, takip edilir.
User(id: "123", name: "Ahmet")

Aggregate Root

Entity kumesinin yoneticisi. Tek giris noktasi.
Is kurallarini korur, disaridan dogrudan ic nesnelere erisilmez.

Domain Service

Birden fazla Root'u koordine eden stateless servis.
Ornek: Para transferi (2 hesap arasi).

Yazilim Mimarileri Software Architectures

Kodun nasil organize edilecegi. Hangi dosya nerede, kim kimi cagirir, bagimliliklar nasil yonetilir.

Monolith

Tek uygulama, tek deploy. Her sey bir arada.
Avantaj: Basit, debug kolay
Dezavantaj: Buyudukce yavaslama, tek hata her seyi cokertirir

Modular Monolith

Tek uygulama ama ic yapi modullere bolunmus. Her modul bagimsiz.
TicketHub bu mimariyi kullaniyor.
Monolith'in basitligi + microservice'in organizasyonu.

Microservices

Her modul ayri uygulama, ayri DB, ayri deploy.
Avantaj: Bagimsiz olcekleme, takim ozerkligi
Dezavantaj: Agir operasyonel yuk, network hatasi

Katmanli Mimari (Layered)

Controller → Service → Repository → DB
Her katmanin tek isi var. Ust katman alt katmani cagirir, tersi olmaz.

Hexagonal (Ports & Adapters)

Domain merkeze, disaridaki her sey (DB, API, UI) takilabilir adaptor.
DB degistirmek = sadece adaptor degistir, domain'e dokunma.

Clean Architecture

Bagimlilk yonu her zaman iceri dogru.
Dis katman ic katmani bilir, ic katman disi bilmez.
Domain → hicbir seye bagimli degil.

Veritabani Kullanimi ve Yonetimi Database Usage & Management

Veri kalici olarak nerede ve nasil saklanir. Backend'in en kritik bileseni — veri kaybedilirse her sey kaybedilir.

SQL vs NoSQL

SQL: Tablolar, iliskiler, sema zorunlu. PostgreSQL, MySQL, SQL Server.
NoSQL: Esnek yapi, yatay olcekleme. MongoDB, Redis, Cassandra.

Ne Zaman Hangisi?

SQL: Iliskisel veri, tutarlilik kritik (banka, e-ticaret)
NoSQL: Esnek sema, yuksek yazma hizi (log, IoT, sosyal medya)

CRUD Islemleri

Create → INSERT INTO
Read → SELECT FROM
Update → UPDATE SET
Delete → DELETE FROM

ACID

Atomicity: Ya hepsi ya hicbiri
Consistency: Veri her zaman gecerli
Isolation: Islemler birbirini etkilemez
Durability: Kaydedilen kaybolmaz

SQL Temelleri

-- Tablo olustur
CREATE TABLE users (
    id       INT PRIMARY KEY,
    name     VARCHAR(100) NOT NULL,
    email    VARCHAR(255) UNIQUE NOT NULL,
    age      INT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Veri ekle
INSERT INTO users (name, email, age) VALUES ('Ahmet', 'ahmet@mail.com', 25);

-- Sorgula
SELECT name, email FROM users WHERE age > 18 ORDER BY name;

-- Guncelle
UPDATE users SET age = 26 WHERE id = 1;

-- Sil
DELETE FROM users WHERE id = 1;

JOIN — Tablolari Birlestirme

-- Kullanicinin siparislerini getir (2 tablo birlestir)
SELECT u.name, o.total, o.status
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.status = 'CONFIRMED';

-- JOIN turleri:
-- INNER JOIN  → iki tabloda da eslesen kayitlar
-- LEFT JOIN   → sol tablodaki tum + sag tabloda eslesen
-- RIGHT JOIN  → sag tablodaki tum + sol tabloda eslesen
-- FULL JOIN   → her iki tablodaki tum kayitlar

Index — Sorgulari Hizlandirma

Index Nedir?

Kitabin arkasindaki dizin gibi. Index olmadan tum tabloyu tarar (O(n)). Index ile aninda bulur (O(log n)).
CREATE INDEX idx_email ON users(email);
Dezavantaj: Yazma yavaslar (her INSERT'te index de guncellenir). Sadece sik sorgulanan alanlara ekle.

Veritabani Yonetimi Database Administration

DB'yi canlida yonetmek: performans, yedekleme, goc, replikasyon.

Migration (Goc)

DB semasini kod ile versiyonla. Tablo ekle, alan degistir — hepsi kayitli.
prisma migrate dev veya dotnet ef migrations add
Git gibi ama DB icin.

Backup & Recovery

Duzenii yedekleme: pg_dump (PostgreSQL)
Point-in-time recovery: belirli bir ana geri don.
Kural: Yedeklenmeyen veri yoktur.

Replikasyon

Primary → Replica: Yazma primary'e, okuma replica'dan.
Neden? Okuma yogun sistemlerde yuku dagit.
Primary cokerse replica devralir (failover).

Connection Pool

Her istek icin yeni DB baglantisi acmak pahali (~5ms). Pool: onceden N baglanti ac, tekrar kullan.
pool_size: 20 → max 20 eszamanli DB baglantisi.

Normalizasyon

Veri tekrarini onlemek icin tablolari bolme:
1NF: Tekrarlayan grup yok
2NF: Kismi bagimlilik yok
3NF: Gecisli bagimlilik yok
Pratikte 3NF yeterli.

Transaction

Birden fazla islemi tek atom haline getir:
BEGIN → islem1 → islem2 → COMMIT
Biri basarisiz → ROLLBACK → hepsi geri alinir.

ORM (Prisma, EF Core) SQL'i Gizler — Ama Bilmen Gerekir

ORM kolaylastirma aracii. Altinda SQL calisiyor. Performans sorunu ciktiginda, mulaakatta veya debug sirasinda SQL bilmek ZORUNLU. ORM'e guvenip SQL'i atlama.

// C# .NET Core — Controller ornegi
[ApiController]
[Route("api/v1/[controller]")]
public class UsersController : ControllerBase
{
    private readonly UserService _service;

    public UsersController(UserService service) => _service = service;

    [HttpGet]
    public async Task<IActionResult> GetAll([FromQuery] int page = 1, [FromQuery] int size = 20)
    {
        var result = await _service.GetPagedAsync(page, size);
        return Ok(result);
    }

    [HttpGet("{id}")]
    public async Task<IActionResult> GetById(int id)
    {
        var user = await _service.GetByIdAsync(id);
        if (user == null) return NotFound();
        return Ok(user);
    }

    [HttpPost]
    [Authorize]  // JWT token gerekli
    public async Task<IActionResult> Create([FromBody] CreateUserRequest request)
    {
        var user = await _service.CreateAsync(request);
        return CreatedAtAction(nameof(GetById), new { id = user.Id }, user);
    }
}

Guvenli Yazilim Gelistirmenin Temelleri Secure Software Development

Guvenlik sonradan eklenen bir ozellik degil, tasarimdan itibaren dusunulmesi gereken temel prensip.

SQL Injection

Kullanici girdisi SQL'e dogrudan eklenir:
WHERE name = '" + input + "' ❌
Input: ' OR 1=1 -- → tum verileri getirir
Cozum: Parameterized query kullan.

XSS (Cross-Site Scripting)

Kullanicinin girdigi JS kodu sayfada calisir:
Yorum: <script>steal(cookie)</script>
Cozum: HTML encode, Content-Security-Policy.

CSRF (Cross-Site Request Forgery)

Kullanicinin oturumunu kullanarak istek gonderme.
Baska sitedeki link: POST /transfer?amount=1000
Cozum: CSRF token, SameSite cookie.

Input Validation

Kullanicidan gelen HER veriyi dogrula:
Tip kontrolu, uzunluk limiti, format kontrolu.
Kural: Kullanici girdisine ASLA guvenmee.

Sifre Yonetimi

Sifreyi asla duz metin saklama.
bcrypt veya Argon2 ile hashle.
Minimum 12 karakter, karmasiklik zorunlu.
